📖 Glossaire des Concepts Clés

UML (Unified Modeling Language)
Langage visuel standardisé pour modéliser les systèmes logiciels. UML est un langage, pas une méthode.
MDA (Model Driven Architecture)
Architecture dirigée par les modèles, approche de développement logiciel basée sur la transformation de modèles à différents niveaux d'abstraction.
CIM (Computation Independent Model)
Modèle indépendant de l'informatique qui décrit le domaine métier sans considération technique. Répond aux questions : Qui ? Quoi ? Quelles activités ?
PIM (Platform Independent Model)
Modèle indépendant de la plateforme qui représente la logique de la solution avec des éléments techniques (UUID, opérations) sans choix technologique spécifique.
PSM (Platform Specific Model)
Modèle spécifique à une plateforme qui adapte le PIM à une technologie concrète (PostgreSQL, Java, Docker, etc.).
XMI (XML Metadata Interchange)
Format d'échange standard XML pour les modèles UML, permettant l'interopérabilité entre outils de modélisation professionnels.
MBE (Model-Based Engineering)
Ingénierie basée sur les modèles, approche où les modèles sémantiques sont la source unique de vérité pour le développement.
PlantUML
Outil de génération de diagrammes à partir de syntaxe textuelle. Important : génère des dessins (images), pas des modèles sémantiques exploitables.
Paradigme Objet
Approche de modélisation basée sur les classes, l'encapsulation, l'héritage et le polymorphisme. Définit la structure (attributs) et le comportement (opérations).
Paradigme Composant
Évolution du paradigme objet pour les systèmes distribués. Met l'accent sur la séparation des préoccupations, la modularité, l'interopérabilité et les interfaces contractuelles.
Pensée Systémique
Approche holistique pour comprendre les systèmes complexes basée sur 6 principes : intégrité, interaction, rétroaction, conditions initiales, émergence et adaptation.

❓ Questions auxquelles il faut savoir répondre

Q1 : Puis-je utiliser uniquement PlantUML pour mon projet final ?

PlantUML est excellent pour la phase d'exploration initiale et la génération rapide du CIM avec l'aide de ChatGPT. Cependant, pour le PIM et pour une évaluation sérieuse, il est fortement recommandé de passer à Papyrus pour bénéficier de la validation automatique, de la cohérence multi-diagrammes et de la génération de code.

Approche recommandée : Prototypage en PlantUML → Validation et enrichissement en Papyrus → Soumission du modèle Papyrus (XMI) + documentation PlantUML complémentaire.

Q2 : Quelle est la différence concrète entre CIM et PIM ?

CIM (Computation Independent Model) : Décrit "QUOI" - le métier pur sans considération technique.

  • Acteurs métier (Client, Fournisseur)
  • Objets conceptuels (Commande, Facture)
  • Processus métier (passer une commande)
  • Informations échangées (montant, date)
  • Pas d'UUID, pas d'opérations informatiques
PIM (Platform Independent Model) : Décrit "COMMENT logiquement" - l'architecture logicielle abstraite.
  • Classes logiques (ClientEntity, CommandeEntity)
  • Identifiants techniques (uuid:UUID, version:int)
  • Opérations logicielles (validate():boolean, save():void)
  • Architecture en couches (Présentation, Métier, Données)
  • Pas encore de technologie spécifique (pas de PostgreSQL, Java, etc.)
Exemple concret :
  • CIM : Classe "Client" avec attribut "nom"
  • PIM : Classe "ClientEntity" avec "uuid:UUID", "nom:String", "email:String", "createdAt:DateTime", méthode "validate():boolean"
  • PSM : Table PostgreSQL "clients" avec colonnes spécifiques + contraintes SQL

Q3 : Le paradigme composant remplace-t-il l'approche objet ?

Non, il la complète et l'étend. L'approche objet reste valide et pertinente pour la structure interne des composants (encapsulation, héritage local, polymorphisme).

Le paradigme composant s'applique au niveau architectural pour :

  • Gérer les systèmes distribués multi-fournisseurs
  • Assurer l'interopérabilité via interfaces contractuelles
  • Permettre la reconfigurabilité dynamique
  • Éviter le couplage fort de l'héritage au niveau système
En pratique : Vous utilisez des objets (classes) À L'INTÉRIEUR de vos composants, et des composants pour structurer votre architecture globale. C'est complémentaire, pas exclusif.

Q4 : Papyrus est-il obligatoire ou puis-je utiliser un autre outil ?

Papyrus est recommandé car il est gratuit, open source et crée de vrais modèles XMI formels. Vous pouvez utiliser d'autres ateliers professionnels si vous y avez accès :

  • ✅ Modelio (alternative open source)
  • ✅ Enterprise Architect (si licence disponible)
  • ✅ MagicDraw/Cameo (si licence disponible)
L'important est d'utiliser un outil qui manipule des modèles sémantiques (format XMI), pas juste des dessins. Si vous utilisez un autre outil, assurez-vous de pouvoir :
  • Exporter en XMI pour interopérabilité
  • Valider automatiquement le modèle
  • Générer du code depuis le modèle

Q5 : Quels diagrammes UML dois-je absolument maîtriser ?

Pour ce cours, concentrez-vous sur les diagrammes essentiels :

Obligatoires (maîtrise requise) :

  • Diagramme de classes : Structure statique (CIM et PIM). Le plus important.
  • Diagramme de séquence : Comportements dynamiques, chronologie des interactions.

Recommandés (selon domaine) :

  • Diagramme de composants : Architecture modulaire (PIM). Utile pour comprendre le paradigme composant.
  • Diagramme de cas d'utilisation : Exigences fonctionnelles (CIM). Bon pour le contexte.

Optionnels (approfondissement) :

  • Diagramme d'activité (workflows)
  • Diagramme d'états (machines à états)
  • Diagramme de déploiement (PSM infrastructure)
Conseil : Mieux vaut bien maîtriser classes + séquence que survoler tous les types de diagrammes.

Q6 : ChatGPT peut-il faire le travail à ma place ?

Non, absolument pas. ChatGPT est un assistant pour catalyser l'extraction du vocabulaire et générer une première itération, mais :

Vous devez impérativement :

  • ✅ Valider la pertinence des concepts extraits (l'IA peut inventer)
  • ✅ Corriger les erreurs et imprécisions nombreuses
  • ✅ Enrichir le modèle avec votre connaissance métier approfondie
  • ✅ Critiquer et améliorer substantiellement les propositions
  • ✅ Justifier vos choix de modélisation (ça, l'IA ne peut pas le faire)

Limites critiques de l'IA :

  • ❌ Hallucine fréquemment (invente des concepts inexistants)
  • ❌ N'a pas la compréhension fine du contexte métier réel
  • ❌ Ne peut pas valider la pertinence opérationnelle
  • ❌ Génère souvent des erreurs de syntaxe PlantUML
  • ❌ Ne comprend pas les enjeux organisationnels
Votre valeur ajoutée irremplaçable : Vous êtes l'expert métier, garant de la validité du modèle. L'IA est un copilote, pas un pilote. C'est VOTRE expertise qui crée de la valeur et qui sera évaluée dans l'entreprise, la connaissance et le savoir faire acquis à l'université.

En évaluation : Nous vérifierons la profondeur de compréhension et la capacité à justifier. Un modèle "copié-collé" de ChatGPT sans compréhension sera facilement détectable et mal noté.

Q7 : Que faire si j'ai des problèmes d'installation avec Papyrus ?

Solutions par plateforme :

macOS (M1/M2/M3) :

  • Problème de compatibilité ARM connu
  • Solution 1 : Utiliser la version Intel avec Rosetta
  • Solution 2 : Attendre version native ARM (à venir)
  • Solution 3 : Utiliser Modelio (alternative) ou machine virtuelle Linux

Windows 11 :

  • Vérifier Java : java -version doit afficher JRE 11+
  • Si absent : installer OpenJDK depuis adoptium.net
  • Relancer Papyrus après installation Java
  • Si lenteur : allouer plus de RAM (éditer papyrus.ini, augmenter -Xmx)

Linux :

  • Installer via gestionnaire : sudo apt install openjdk-11-jre
  • Vérifier permissions exécution : chmod +x papyrus
  • En général excellent support Linux

Alternative universelle :

  • Essayer Modelio (autre outil open source MDA)
  • Ou utiliser Eclipse standard + installer plugin Papyrus
  • Ou utiliser une version antérieure plus stable (7.1 semble instable -> utiliser 6.7 si bloqué)
Support : Partager votre problème dans le chat du cours ou par email. La communauté peut aider.

Q9 : Pourquoi insister autant sur "Modéliser ≠ Dessiner" ?

Parce que c'est la distinction fondamentale qui différencie l'ingénierie professionnelle du bricolage, et c'est le cœur de ce cours.

Impact concret :

Avec du dessin (PlantUML, PowerPoint) :

  • ❌ Modifier "Client" → chercher et corriger manuellement dans 15 fichiers
  • ❌ Ajouter attribut → risque d'oublier certains diagrammes → incohérence
  • ❌ Générer code → impossible ou manuel et très fragile
  • ❌ Vérifier cohérence → relecture humaine fastidieuse et sujette à erreurs
  • ❌ Faire évoluer → maintenance cauchemardesque

Avec un modèle formel (Papyrus, MagicDraw) :

  • ✅ Modifier "Client" → propagation automatique dans TOUS les diagrammes
  • ✅ Ajouter attribut → visible partout immédiatement
  • ✅ Générer code → automatique, fiable, synchronisable
  • ✅ Vérifier cohérence → validation automatique en temps réel
  • ✅ Faire évoluer → maintenance simplifiée, refactoring sûr

Exemple industriel réel :

Dans l'aéronautique (Airbus, Boeing), un système comme le contrôle de vol peut avoir des centaines de classes et milliers de relations. Maintenir ça avec PowerPoint ou PlantUML serait tout simplement impossible. D'où l'utilisation obligatoire d'outils professionnels (MagicDraw, Rhapsody) avec modélisation formelle.


Ce cours vous prépare à cette réalité industrielle. Comprendre la différence dessin/modélisation vous donne une compétence professionnelle valorisable.

📅 Session 03 : Pensée Systémique et Paradigme Composant

🎯 Objectifs d'apprentissage

  • Maîtriser les six principes de la théorie des systèmes
  • Comprendre les limites de l'approche orientée objet pour les systèmes distribués
  • Saisir les avantages du paradigme composant (séparation des préoccupations, modularité, interopérabilité)
  • Appliquer la pensée systémique à la conception de systèmes complexes

Résumé

La session a introduit l'importance de la pensée systémique et de la modélisation pour traiter des systèmes complexes. Les principes clés de la théorie des systèmes ont été présentés, ainsi que l'évolution des paradigmes de développement logiciel, du modèle orienté objet vers les approches basées sur les composants, mises en avant pour leur flexibilité et leur reconfigurabilité dans les systèmes distribués.

1. Fondamentaux de la pensée systémique

L'intervenant a mis en évidence l'importance de la modélisation dans les activités d'ingénierie et a introduit la notion de pensée systémique, en la comparant à la pensée analytique. L'accent a été mis sur la valeur des approches holistiques pour appréhender la complexité des systèmes. Des exemples issus du domaine aéronautique ont illustré comment la modélisation contribue à réduire les coûts et accélérer la mise sur le marché.

2. Les six principes clés de la théorie des systèmes

Principe Description Exemple
Intégrité Le système fonctionne comme un tout cohérent Un avion n'est pas une somme de pièces mais un système intégré
Interaction Les composants interagissent entre eux Les sous-systèmes communiquent via des interfaces
Rétroaction Les sorties influencent les entrées (boucles de contrôle) Thermostat régulant la température
Conditions initiales L'état initial détermine l'évolution du système Configuration de départ d'un projet
Émergence Propriétés nouvelles apparaissent au niveau système La conscience émerge du cerveau
Adaptation Le système évolue avec son environnement Systèmes auto-adaptatifs

3. Conception de systèmes avec rétroaction

L'exploration a porté sur la conception de systèmes intégrant la rétroaction, le contrôle et le comportement. Le formateur a souligné la nécessité d'architectures alignées sur les performances attendues. Discussion sur la construction, la maintenance, le recyclage et la réduction d'impact énergétique. Une distinction a été faite entre approches systémiques dures (quantitatives, ingénierie) et souples (qualitatives, organisationnelles).

4. Transition vers le développement logiciel basé sur des composants

L'évolution des paradigmes logiciels a été abordée, insistant sur le passage de la programmation orientée objet aux approches à base de composants. Il a été montré que le paradigme objet devient limité pour les systèmes distribués, notamment en raison du couplage fort et des problèmes d'héritage.

⚠️ Limites de l'approche objet pour les systèmes distribués

  • Couplage fort : L'héritage crée des dépendances rigides difficiles à faire évoluer
  • Fragile Base Class Problem : Modifier une classe mère casse les classes filles
  • Pas d'interopérabilité : Difficile d'intégrer des composants de fournisseurs différents
  • Complexité croissante : Les hiérarchies d'héritage deviennent ingérables
Avantages de l'approche composant : Séparation des préoccupations, modularité, interopérabilité via interfaces normalisées, reconfigurabilité dynamique et standardisation. Les composants sont développés indépendamment puis assemblés via des contrats d'interface.

5. Évolution vers des systèmes ouverts et interopérables

La fin de la session a porté sur la transformation des systèmes fermés vers des systèmes ouverts et interopérables. Explication de la manière dont des composants peuvent être développés indépendamment puis intégrés grâce à des interfaces normalisées. Des analogies avec le bâtiment (prises électriques standardisées) ou l'aéronautique ont illustré la logique contractuelle de ces architectures.

📅 Session 04 : Ingénierie par les Modèles (MDA) - Modéliser ≠ Dessiner

🎯 Objectifs d'apprentissage

  • Maîtriser les trois niveaux MDA : CIM, PIM, PSM et leur articulation
  • Distinguer clairement modélisation sémantique et dessin graphique
  • Utiliser les diagrammes de classes et de séquence à bon escient selon le niveau d'abstraction
  • Comprendre le rôle et les limites strictes de PlantUML
  • Positionner son rôle d'expert métier dans la chaîne de valeur du développement

Vue d'ensemble

Cette session constitue un tournant décisif dans l'enseignement de l'UML et de l'ingénierie par les modèles. Elle introduit l'Ingénierie par les Modèles (MDA) et l'utilisation d'UML, en soulignant la distinction fondamentale entre modéliser et simplement dessiner. L'approche s'organise selon trois niveaux essentiels qui structurent toute la démarche de développement.

Les trois niveaux de modélisation MDA

Niveau Description Contenu typique Acteur principal Questions clés
CIM
(Computation Independent Model)		 Vocabulaire et concepts métier purs Acteurs, artefacts métier, processus, informations échangées Expert métier Qui ? Quoi ? Quelles activités ? Quelles informations ?
PIM
(Platform Independent Model)		 Architecture logique de la solution Classes avec UUID, opérations techniques, composants logiques, interfaces Architecte logiciel Comment (logiquement) ? Quel paradigme ? Quelle structure ?
PSM
(Platform Specific Model)		 Implémentation technique ciblée Tables PostgreSQL, classes Java, conteneurs Docker, APIs REST Développeur / Architecte technique Avec quelle technologie ? Quel framework ? Quelle plateforme ?

💡 Principe de séparation des préoccupations

Cette séparation stricte entre le "quoi" (CIM), le "comment logique" (PIM) et le "comment technique" (PSM) est le fondement d'une ingénierie logicielle robuste et maintenable. Elle permet :

  • De faire évoluer la technologie sans remettre en cause le métier
  • De réutiliser le PIM pour différentes plateformes (web, mobile, embarqué)
  • De garantir la traçabilité des exigences métier vers l'implémentation
  • De faciliter la communication entre experts de domaines différents

Diagrammes UML : Dualité d'usage selon le niveau

Le Diagramme de Classes

Le diagramme de classes illustre parfaitement la dualité d'usage d'un même type de diagramme :

Utilisation Niveau Contenu Exemple
Vocabulaire métier CIM Classes conceptuelles, attributs métier, associations sémantiques Classe "Client" avec attribut "nom" (pas d'UUID, pas d'opérations techniques)
Structure logicielle PIM Classes logiques avec UUID, version, opérations (validate(), execute()) Classe "ClientEntity" avec uuid:UUID, opération validate():boolean

La différence réside dans le contenu sémantique des classes, non dans la syntaxe graphique. Un même diagramme de classes peut servir à deux usages totalement différents.

Le Diagramme de Séquence

Outil privilégié pour capturer les comportements dynamiques. Il impose un ordre chronologique (lecture de haut en bas) et matérialise les échanges de messages entre acteurs ou objets. Fonctionne comme une "recette de cuisine" où les flèches horizontales représentent les appels d'opérations.

Expressivité : Permet de modéliser des boucles, des conditions, des appels parallèles et même des sous-séquences. C'est un langage procédural complet qui peut être simulé.

Outils : La distinction cruciale entre Dessin et Modèle Sémantique

⚠️ CRUCIAL : PlantUML est un outil de DESSIN, pas de modélisation

PlantUML génère des images (SVG, PNG), pas des modèles sémantiques exploitables par la machine.

Il est utile pour :

  • ✅ Prototypage rapide et exploration initiale d'un domaine
  • ✅ Documentation visuelle statique intégrée au code (Markdown)
  • ✅ Communication informelle des concepts
  • ✅ Génération rapide avec l'aide de LLM (ChatGPT)
  • ✅ Versionnement avec Git (format texte)

Mais il ne permet PAS :

  • ❌ Validation automatique de la cohérence du modèle
  • ❌ Vérification des règles UML
  • ❌ Génération de code fiable et maintenable
  • ❌ Navigation entre diagrammes liés
  • ❌ Manipulation sémantique des éléments (refactoring)
  • ❌ Traçabilité entre niveaux (CIM → PIM → PSM)
  • ❌ Cohérence multi-diagrammes garantie

Pourquoi ? Parce que le code SVG/PNG généré ne contient que des primitives graphiques (<rect>, <text>, <line>). L'ordinateur voit des rectangles, pas des concepts UML identifiés avec leurs propriétés et relations.

Recommandation pédagogique : Utilisez PlantUML pour démarrer rapidement et explorer vos idées, puis passez impérativement à un outil de modélisation professionnel (Papyrus, MagicDraw, Enterprise Architect) pour tout travail critique nécessitant validation, cohérence et génération.

Outils Professionnels : La Vraie Modélisation

Les outils professionnels (MagicDraw, Papyrus, Enterprise Architect, Modelio) manipulent des modèles sémantiques stockés en format XMI (XML Metadata Interchange). Chaque élément possède un identifiant unique et des métadonnées complètes.

Capacités essentielles :

  • Validation automatique des règles UML
  • Cohérence garantie entre tous les diagrammes
  • Refactoring automatique (renommer une classe met à jour toutes ses références)
  • Génération de code vers multiples langages
  • Reverse engineering (code → modèle)
  • Simulation et vérification formelle
  • Traçabilité complète (exigences → modèle → code)
  • Gestion de versions du modèle

Rôle de l'IA et positionnement dans la chaîne de valeur

Les LLM comme assistants de formalisation

L'originalité pédagogique de cette session réside dans l'utilisation des Large Language Models (ChatGPT, Claude) pour catalyser la formalisation du vocabulaire métier. Le processus recommandé :

  1. Décrire son domaine métier en langage naturel au LLM
  2. Demander l'extraction du vocabulaire et des concepts principaux
  3. Faire générer un CIM en code PlantUML
  4. Critiquer et enrichir le résultat (l'IA peut se tromper !)
  5. Itérer jusqu'à obtenir un CIM satisfaisant
  6. Demander la génération du PIM correspondant
  7. Recréer le modèle dans un outil professionnel pour validation

⚠️ Limites critiques de l'IA générative

L'IA ne remplace PAS l'expertise humaine

  • Les LLM peuvent halluciner (inventer des concepts inexistants)
  • Ils n'ont pas la compréhension fine du contexte métier
  • Ils ne peuvent pas valider la pertinence opérationnelle
  • Ils génèrent souvent des erreurs de syntaxe PlantUML

Votre rôle d'expert métier est irremplaçable : Vous êtes le garant de la validité du modèle, capable de critiquer, corriger et enrichir les propositions de l'IA. C'est VOTRE expertise qui crée de la valeur.

Positionnement dans la chaîne de développement

L'objectif pédagogique n'est pas de former des analystes professionnels, mais de rendre les futurs ingénieurs (hydrographes, automaticiens, etc.) capables de :

  • Formaliser leur expertise métier en modèles compréhensibles
  • Dialoguer efficacement avec les analystes et concepteurs
  • Critiquer et valider les modèles proposés
  • Garantir que les applications répondent aux besoins opérationnels réels
  • Communiquer les besoins aux informaticiens sans ambiguïté
Message central : "Modéliser n'est pas dessiner". Un modèle est composé d'objets typés identifiés manipulables avec validation automatique, tandis qu'un dessin est une image statique sans sémantique exploitable par la machine. Cette distinction est le cœur de l'ingénierie par les modèles.

Comparatif détaillé des outils de représentation

1. Solutions Textuelles (PlantUML, Mermaid)

Principe : Génération de diagrammes à partir de syntaxe textuelle avec layout automatique.

Avantages :

  • ✅ Rapidité : Génération quasi-instantanée
  • ✅ Layout automatique : Pas de positionnement manuel
  • ✅ Versionnement : Intégration parfaite avec Git
  • ✅ Documentation intégrée : Diagrammes dans le code Markdown
  • ✅ Assistance IA : Génération facile via LLM
  • ✅ Gratuit et open source

Limites critiques :

  • ❌ Syntaxe non-standard : Chaque outil a sa propre syntaxe
  • ❌ Modification laborieuse : Une faute de frappe sur un nom détruit toutes les relations
  • ❌ Absence d'interactivité : Impossible de filtrer dynamiquement
  • ❌ Pas de vérification sémantique : L'outil ne comprend pas UML
  • ❌ Manipule du code, pas un modèle formel
  • ❌ Aucune cohérence multi-diagrammes garantie

💡 Problème concret de PlantUML

Si vous écrivez une classe "Transformation" et que vous faites une faute en l'appelant "Transformaton" dans une relation, PlantUML crée silencieusement une NOUVELLE classe. Vous devez corriger manuellement toutes les occurrences dans le texte. Un outil de modélisation professionnel aurait signalé l'erreur immédiatement.

Usage recommandé : Prototypage rapide, exploration initiale, documentation technique légère, communication informelle.

2. Outils de Dessin avec Layout Automatique (yEd, Graphviz)

Principe : Éditeurs graphiques avec algorithmes de mise en page sophistiqués.

Avantages :

  • ✅ Interface visuelle intuitive : Glisser-déposer direct
  • ✅ Algorithmes de layout puissants : Organique, hiérarchique, circulaire, orthogonal
  • ✅ Manipulation interactive : Colorisation, regroupement, filtrage visuel
  • ✅ Export multi-formats : PDF, PNG, SVG, GraphML
  • ✅ Gratuit

Limite majeure :

  • ❌ Format GraphML : Contient uniquement des primitives géométriques (<node>, <edge>)
  • ❌ Pas de concepts UML natifs : Pas de notion de Classe, Attribut, Opération
  • ❌ Validation humaine uniquement : L'ordinateur ne "comprend" pas le diagramme
  • ❌ C'est un outil de dessin, pas de modélisation

⚠️ Exemple : Analyse du code GraphML généré par yEd

Le fichier XML ne contient que :

<node id="n1">
  <geometry x="100" y="50" width="120" height="80"/>
  <fill color="#FFCC00"/>
  <label>Transformation</label>
</node>

L'ordinateur voit un rectangle jaune avec du texte, PAS une classe UML avec ses propriétés et comportements. La sémantique existe uniquement dans l'interprétation humaine.

Usage recommandé : Diagrammes esthétiques pour présentations, graphes génériques, visualisation de données relationnelles.

3. Ateliers de Modélisation Professionnels (MDA Tools)

Outils : Enterprise Architect, MagicDraw/Cameo, Modelio, Papyrus, Rational Software Architect

Principe fondamental : Manipulation de modèles sémantiques formels stockés en format XMI (XML Metadata Interchange). Chaque élément UML possède un identifiant unique et des métadonnées complètes.

Capacités essentielles :

  • ✅ Modèle unique, vues multiples : Un seul modèle, N diagrammes comme projections
  • ✅ Cohérence automatique multi-diagrammes : Modifier un élément met à jour toutes ses vues
  • ✅ Validation automatique : Respect des règles et contraintes UML
  • ✅ Génération de code : Transformation modèle → code (Java, C++, Python, SQL...)
  • ✅ Reverse engineering : Code → modèle (synchronisation bidirectionnelle)
  • ✅ Refactoring sémantique : Renommer propage partout automatiquement
  • ✅ Simulation et vérification : Exécution de modèles pour validation
  • ✅ Traçabilité complète : Exigences → modèle → code
  • ✅ Gestion de versions du modèle : Diff et merge intelligents
  • ✅ Intégration ALM : Jira, Git, Jenkins, gestion d'exigences
  • ✅ Profils UML : Extension pour domaines spécifiques (SysML, MARTE...)
  • ✅ Interopérabilité : Import/export XMI entre outils
Le modèle est la source unique de vérité (structure de données arborescente). Le diagramme est une vue partielle pour la communication. Seuls les ateliers professionnels garantissent cette séparation et permettent l'ingénierie par les modèles (MBE).

Philosophie : Un modèle unique, multiples vues

💡 Principe fondamental de la modélisation visuelle

La modélisation visuelle est basée sur un modèle unique (structure de données arborescente contenant tous les éléments UML) dont les diagrammes multiples sont de simples vues (projections visuelles partielles).

Garanties offertes :

  • ✅ Cohérence automatique : Modifier le nom d'une classe dans le modèle propage le changement dans TOUS les diagrammes automatiquement
  • ✅ Validation automatique : Respect des règles UML vérifié en temps réel
  • ✅ Génération de code : Depuis le modèle (source de vérité), pas depuis un diagramme particulier
  • ✅ Interopérabilité : Export/import en format XMI standard
  • ✅ Pas de duplication : Un élément existe une seule fois dans le modèle

⚠️ Conséquence du dessin statique

Avec PowerPoint ou PlantUML, si vous renommez une classe, vous devez :

  • Modifier manuellement chaque diagramme où elle apparaît
  • Risque élevé d'oublier des occurrences → incohérence
  • Aucune validation automatique

Avec Papyrus : une seule modification dans le modèle, propagation automatique partout.

Zone Description Rôle et usage

Création d'éléments : 2 méthodes

Méthode 1 : Depuis le Model Explorer (plus explicite)

  1. Clic droit sur le package dans le Model Explorer
  2. New Child → Class (ou autre type)
  3. L'élément est créé dans le modèle
  4. Glisser-déposer l'élément depuis le Model Explorer vers le diagramme pour le visualiser

Avantage : Contrôle total, compréhension claire de la séparation modèle/vue.

Méthode 2 : Depuis la Palette (plus rapide)

  1. Sélectionner l'outil "Class" dans la Palette
  2. Cliquer dans le diagramme
  3. L'élément est créé simultanément dans le modèle ET dans la vue

Avantage : Rapidité pour la modélisation itérative.

Propriétés : Modèle vs Visuel

Distinction fondamentale :
  • Propriétés du MODÈLE (sémantiques) : Nom, attributs, opérations, visibilité, types, multiplicités, contraintes → Propagées automatiquement dans toutes les vues
  • Propriétés VISUELLES (présentation) : Couleur, position, taille, police, affichage/masquage des sections → Spécifiques à chaque diagramme

Exemple pratique :

Action Type Effet
Renommer la classe "Client" en "Customer" Propriété MODÈLE ✅ Changement propagé dans TOUS les diagrammes automatiquement
Ajouter un attribut "email:String" Propriété MODÈLE ✅ Visible dans tous les diagrammes contenant cette classe
Colorer la classe en jaune Propriété VISUELLE ❌ Spécifique au diagramme actuel uniquement
Masquer les opérations Propriété VISUELLE ❌ N'affecte que la vue actuelle (les opérations restent dans le modèle)

Gestion des vues multiples

Un même objet peut apparaître dans plusieurs diagrammes différents (vue architecture, vue détaillée, vue simplifiée, etc.). C'est une force majeure de la modélisation.

Création de vues multiples

  1. Créer un nouveau diagramme : Clic droit sur le modèle → New Diagram → Class Diagram
  2. Glisser-déposer des éléments existants depuis le Model Explorer
  3. Les éléments apparaissent dans le nouveau diagramme
  4. Chaque diagramme peut avoir un layout et un style différents

CRITIQUE : Opérations de suppression

⚠️ ATTENTION : Deux types de suppression totalement différents

1. "Delete from Diagram" (Suppr ou Delete)

  • Supprime uniquement la représentation visuelle dans le diagramme actuel
  • L'élément reste dans le modèle
  • Il reste visible dans les autres diagrammes
  • Usage : Simplifier la vue, masquer temporairement

2. "Delete from Model" (Clic droit → Delete from Model)

  • Supprime l'objet du MODÈLE (source unique de vérité)
  • Disparaît de TOUS les diagrammes automatiquement
  • Disparaît du Model Explorer
  • Action irréversible (sauf Undo)
  • Usage : Supprimer définitivement un concept du modèle

Piège fréquent : Utiliser "Delete from Diagram" en pensant supprimer l'élément définitivement, alors qu'il reste dans le modèle et les autres vues.

Principe de sélection :
  • 🎯 Besoin : Adaptation outil ↔ objectif (documentation vs ingénierie critique)
  • 🔧 Modélisation formelle : Nécessaire dès que validation, cohérence et génération sont requis
  • 💰 Budget : Open source (Papyrus) parfaitement viable pour académique et PME
  • ⚡ Agilité : PlantUML excellent pour prototypage, puis migration vers outil formel
  • 📈 Échelle : Plus le projet est critique/complexe, plus l'outillage professionnel est indispensable
